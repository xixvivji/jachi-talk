# 커뮤니티 도메인 기록

## 문제 정의

커뮤니티는 지역별 질문, 후기, 생활 정보 공유를 위한 도메인이다.

방 양도글과 달리 거래성 필드가 필요하지 않고, 지역, 카테고리, 제목, 본문 중심으로 동작한다.

그래서 `room_handover_posts`와 분리된 `community_posts` 테이블과 API를 사용한다.

## 카테고리 설계

현재 커뮤니티 카테고리는 enum으로 관리한다.

```txt
QUESTION
REVIEW
NOISE
SAFETY
MAINTENANCE_FEE
FACILITY
RESTAURANT
FREE_TALK
```

선택 이유:

- MVP에서 필요한 카테고리 수가 제한적이다.
- enum을 사용하면 요청 검증과 API 문서화가 단순하다.
- 카테고리별 목록 조회를 빠르게 구현할 수 있다.

트레이드오프:

- 카테고리 추가/수정 시 코드 배포가 필요하다.
- 운영자가 관리자 화면에서 카테고리를 동적으로 관리하기 어렵다.

추후 개선:

- 카테고리 테이블 분리
- 카테고리 표시 순서 관리
- 지역별 활성 카테고리 관리

## 삭제 정책

커뮤니티 게시글 삭제는 hard delete가 아니라 soft delete로 처리한다.

현재 구현:

```txt
hidden = true
```

선택 이유:

- 신고와 관리자 기능이 붙을 때 게시글 이력을 보존할 수 있다.
- 댓글이 붙은 게시글을 실제 삭제하면 댓글 처리 정책이 복잡해질 수 있다.
- 운영자 검토나 복구 가능성을 열어둘 수 있다.

트레이드오프:

- 모든 조회에서 `hidden = false` 조건을 신경 써야 한다.
- 데이터가 계속 남기 때문에 정리 정책이 필요하다.

추후 개선:

- `deleted_at`
- `deleted_by`
- `delete_reason`
- 관리자 복구 기능
- 댓글과 신고 상태 연동

## 수정 정책

수정 API는 PATCH 방식이다.

요청에 포함된 필드만 변경하고, 누락되거나 `null`인 필드는 기존 값을 유지한다.

선택 이유:

- 프론트에서 일부 필드만 수정하기 쉽다.
- 카테고리, 제목, 본문 변경을 하나의 API에서 처리할 수 있다.

트레이드오프:

- 의도적으로 값을 비우는 동작과 변경하지 않는 동작을 구분하기 어렵다.
- 현재는 제목과 본문을 빈 문자열로 바꾸는 요청만 validation으로 막는다.

추후 개선:

- 수정 이력 저장
- 작성자 권한 검증
- 관리자 수정/숨김 정책 분리

## 목록 필터 정책

현재 목록 필터:

```txt
regionId
category
```

선택 이유:

- 지역 기반 커뮤니티이므로 `regionId`는 핵심 필터다.
- 사용자는 질문, 후기, 소음, 치안 등 관심 카테고리별로 글을 탐색할 수 있어야 한다.

추후 개선:

- 키워드 검색
- 인기글 정렬
- 최신 댓글순 정렬
- 조회수/좋아요 기반 정렬
