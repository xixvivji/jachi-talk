# 방 양도글 도메인 기록

## 문제 정의

방 양도글은 자취톡 MVP의 핵심 도메인이다.

일반 커뮤니티 게시글과 달리 보증금, 월세, 관리비, 입주 가능일, 계약 종료일, 방 유형, 임대인 동의 상태 같은 거래성 정보가 필요하다.

그래서 커뮤니티 글과 분리된 `room_handover_posts` 테이블과 API를 사용한다.

## 상태 설계

방 양도글 상태는 현재 세 가지다.

```txt
OPEN        모집중
IN_PROGRESS 대화중
CLOSED      완료
```

선택 이유:

- MVP에서 사용자에게 필요한 상태 흐름이 단순하다.
- 모집중, 대화중, 완료만 있어도 목록 노출과 거래 진행 상태를 표현할 수 있다.
- 상태가 너무 많으면 프론트 UI와 필터가 초기에 복잡해진다.

추후 개선:

- 숨김 상태는 거래 상태와 분리해서 `hidden`으로 관리한다.
- 신고로 인한 관리자 숨김 상태가 필요하면 별도 moderation 상태를 검토한다.
- 만료된 글 자동 전환 정책을 검토한다.

## 삭제 정책

방 양도글 삭제는 hard delete가 아니라 soft delete로 처리한다.

현재 구현:

```txt
hidden = true
```

선택 이유:

- 신고, 운영자 검토, 분쟁 대응에 필요한 게시글 이력을 보존할 수 있다.
- 실수로 삭제된 글을 복구할 가능성을 열어둘 수 있다.
- 관리자 기능과 연결하기 쉽다.

트레이드오프:

- 모든 조회에서 `hidden = false` 조건을 신경 써야 한다.
- 데이터가 계속 남기 때문에 정리 정책이 필요하다.
- 단순 MVP 기준으로는 hard delete보다 구현이 조금 복잡하다.

추후 개선:

- `deleted_at`
- `deleted_by`
- `delete_reason`
- 관리자 복구 기능
- 오래된 숨김 글 archive 정책

## 수정 정책

수정 API는 PATCH 방식이다.

요청에 포함된 필드만 변경하고, 누락되거나 `null`인 필드는 기존 값을 유지한다.

선택 이유:

- 프론트에서 일부 필드만 수정하는 흐름을 만들기 쉽다.
- 전체 필드를 매번 보내지 않아도 된다.
- 방 양도글 필드가 많기 때문에 부분 수정이 더 실용적이다.

트레이드오프:

- 의도적으로 값을 비우는 동작과 변경하지 않는 동작을 구분하기 어렵다.
- 현재는 `maintenanceFee` 같은 nullable 필드를 명시적으로 null로 초기화할 수 없다.

추후 개선:

- 필드 초기화가 필요한 경우 별도 요청 정책 검토
- 수정 이력 저장 검토
- 작성자 권한 검증 추가

## 목록 필터 정책

현재 목록 필터:

```txt
regionId
minRent
maxRent
```

선택 이유:

- MVP에서 사용자가 방을 찾을 때 가장 먼저 필요한 조건은 지역과 월세 범위다.
- 조건이 단순하므로 JPQL 기반 repository 쿼리로 충분하다.

트레이드오프:

- 조건이 늘어나면 JPQL이 길어지고 관리가 어려워질 수 있다.
- 동적 정렬, 키워드 검색, 날짜 조건까지 붙으면 현재 방식이 복잡해진다.

추후 개선:

- 보증금 범위 필터
- 입주 가능일 필터
- 방 유형 필터
- 임대인 동의 상태 필터
- 키워드 검색
- QueryDSL 또는 Specification 도입

## 인증 도입 전 임시 정책

현재는 인증이 붙기 전이므로 작성 API에서 `authorId`를 요청으로 받는다.

이 방식은 개발용 임시 구조다.

추후 인증 도입 후:

- 요청 본문의 `authorId` 제거
- 로그인 사용자 정보를 SecurityContext에서 조회
- 작성자 본인만 수정, 삭제, 상태 변경 가능하도록 검증
- 관리자 권한은 별도 정책으로 분리
