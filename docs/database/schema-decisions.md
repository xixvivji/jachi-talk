# DB 설계 의사결정 기록

## 문제 정의

자취톡은 단순 게시판이 아니라 지역 기반 방 양도와 커뮤니티를 함께 다루는 서비스다.

초기 MVP에서 필요한 핵심 흐름은 다음과 같다.

- 사용자가 OAuth로 로그인한다.
- 사용자가 특정 지역에 방 양도글과 커뮤니티 글을 작성한다.
- 사용자는 게시글에 댓글을 남기고, 부적절한 게시글이나 댓글을 신고할 수 있다.
- 서비스는 공공데이터와 이벤트 로그를 활용해 이후 지역 정보, 운영 지표, 관리자 기능으로 확장한다.

따라서 DB는 사용자, OAuth 계정, 지역, 방 양도글, 커뮤니티 글, 댓글, 신고, 공공데이터 스냅샷, 이벤트 로그를 분리해서 설계했다.

## 전체 설계 방향

이번 설계는 MVP를 빠르게 구현하면서도 이후 기능 확장을 크게 막지 않는 방향을 선택했다.

- 서비스 내부 사용자는 `users`를 기준으로 관리한다.
- OAuth 로그인 수단은 `oauth_accounts`로 분리한다.
- 지역은 `regions`로 정규화하고, 게시글은 `region_id`를 참조한다.
- 방 양도글과 커뮤니티 글은 성격이 달라 별도 테이블로 분리한다.
- 댓글과 신고는 여러 대상에 붙을 수 있으므로 `type + id` 방식의 다형 참조로 둔다.
- 공공데이터는 원본 응답을 스냅샷 형태로 저장한다.
- 이벤트 로그는 운영/분석 확장을 위해 별도 테이블로 둔다.

## 테이블별 설계 이유

### users

`users`는 서비스 내부 사용자를 표현한다.

OAuth 제공자의 사용자 ID가 아니라, 서비스 내부에서 글 작성자, 댓글 작성자, 신고자, 관리자 권한을 연결하기 위한 기준 테이블이다.

장점:

- OAuth 제공자가 바뀌거나 늘어나도 서비스 내부 사용자 ID는 유지된다.
- 게시글, 댓글, 신고, 로그가 모두 동일한 사용자 기준으로 연결된다.
- `role`을 통해 일반 사용자와 관리자 권한을 구분할 수 있다.

트레이드오프:

- OAuth만 쓰는 서비스라면 `users`와 `oauth_accounts` 분리가 초기에 다소 복잡해 보일 수 있다.
- 로그인 처리 시 OAuth 계정 조회 후 내부 사용자 조회가 필요하다.

추후 개선:

- 사용자 탈퇴 상태 추가
- 마지막 로그인 시간 추가
- 닉네임 변경 이력 관리
- 정지/차단 상태 추가

### oauth_accounts

`oauth_accounts`는 카카오, 네이버 같은 외부 로그인 계정을 저장한다.

`users`에 OAuth 정보를 직접 넣지 않고 분리한 이유는 한 사용자가 여러 OAuth 계정을 연결하거나, OAuth 제공자가 추가될 가능성을 고려했기 때문이다.

장점:

- 카카오, 네이버, 구글 등 provider 확장이 쉽다.
- `(provider, provider_user_id)` unique 제약으로 OAuth 계정 중복 가입을 막을 수 있다.
- 서비스 내부 사용자와 외부 인증 수단의 역할이 분리된다.

트레이드오프:

- 로그인 시 조인이 필요하다.
- MVP 기준으로는 테이블이 하나 더 늘어난다.

추후 개선:

- 계정 연결/해제 기능
- provider별 이메일 동의 여부 저장
- refresh token 저장 여부 검토

### regions

`regions`는 서비스의 지역 기준 데이터를 저장한다.

자취톡은 지역 기반 방 양도와 동네 커뮤니티가 핵심이므로, 지역명을 게시글마다 문자열로 저장하지 않고 별도 테이블로 분리했다.

장점:

- 지역명 중복과 오타를 줄일 수 있다.
- 법정동 코드 기반 공공데이터 API와 연결하기 쉽다.
- 지역별 게시글 조회, 커뮤니티 조회, 공공데이터 조회가 일관된다.
- 위도/경도를 통해 이후 지도나 거리 기반 기능으로 확장할 수 있다.

트레이드오프:

- 초기 지역 seed 데이터가 필요하다.
- 법정동과 행정동의 차이를 정교하게 다루려면 구조 보강이 필요하다.

추후 개선:

- 법정동/행정동 분리
- 지하철역, 대학교, 상권 등 지역 보조 테이블 추가
- 지역 검색 자동완성
- 좌표 기반 거리 검색

### room_handover_posts

`room_handover_posts`는 방 양도 게시글을 저장한다.

방 양도글은 보증금, 월세, 관리비, 입주 가능일, 계약 종료일, 방 유형, 층수, 반려동물 가능 여부, 임대인 동의 상태처럼 거래성 필드가 필요하다. 커뮤니티 글과 필드 성격이 다르기 때문에 별도 테이블로 분리했다.

장점:

- 방 양도 도메인에 필요한 필드를 명확하게 표현할 수 있다.
- 지역, 상태, 가격 조건 기반 필터링이 쉽다.
- 이후 방 옵션, 이미지, 찜, 거래 상태 변경 기능으로 확장하기 좋다.

트레이드오프:

- 커뮤니티 글과 공통 게시글 기능이 일부 중복될 수 있다.
- 게시글 통합 검색을 만들 때 두 테이블을 함께 조회해야 한다.

추후 개선:

- 이미지 테이블 추가
- 방 옵션 별도 테이블 분리
- 조회수, 찜 수, 문의 수 추가
- 검색 조건이 늘어나면 QueryDSL 또는 Specification 도입
- soft delete용 `deleted_at`, `deleted_by`, `delete_reason` 추가

### community_posts

`community_posts`는 동네 커뮤니티 게시글을 저장한다.

커뮤니티 글은 방 양도글과 다르게 카테고리, 제목, 본문 중심이다. 방 양도글과 하나의 `posts` 테이블로 합치면 nullable 컬럼이 많아지고 도메인 의미가 흐려질 수 있어 분리했다.

장점:

- 커뮤니티 글의 구조가 단순하고 명확하다.
- 카테고리와 지역 기반 조회가 쉽다.
- 방 양도글과 다른 정책을 적용하기 쉽다.

트레이드오프:

- 공통 게시글 기능이 늘어나면 중복 구현이 생길 수 있다.
- 전체 게시글 통합 피드를 만들 때 별도 조회 전략이 필요하다.

추후 개선:

- 게시글 숨김 상태 추가
- 조회수, 좋아요 수 추가
- 공지글, 인기글 정책 추가
- 카테고리 테이블 분리 여부 검토

### comments

`comments`는 댓글을 저장한다.

댓글은 방 양도글에도 달릴 수 있고 커뮤니티 글에도 달릴 수 있으므로 `post_type`, `post_id`를 사용하는 다형 참조 구조로 설계했다.

장점:

- 댓글 테이블 하나로 여러 게시글 타입을 처리할 수 있다.
- 새로운 게시글 타입이 생겨도 댓글 테이블을 추가하지 않아도 된다.
- 댓글 목록 API를 공통화하기 쉽다.

트레이드오프:

- DB 레벨에서 `post_id`에 직접 foreign key를 걸기 어렵다.
- 대상 게시글 존재 여부를 애플리케이션 서비스 로직에서 검증해야 한다.
- 게시글 삭제/숨김 시 댓글 처리 정책을 코드에서 명확히 관리해야 한다.

추후 개선:

- 댓글 숨김 상태 추가
- 대댓글 구조 추가
- 댓글 수 캐싱
- 대상 게시글 검증 로직 강화

### reports

`reports`는 신고 정보를 저장한다.

신고 대상은 방 양도글, 커뮤니티 글, 댓글이 될 수 있으므로 `target_type`, `target_id` 구조를 선택했다.

장점:

- 관리자 신고 목록을 한 테이블에서 조회할 수 있다.
- 신고 대상 타입이 늘어나도 테이블 구조를 크게 바꾸지 않아도 된다.
- 신고 처리 상태를 일관되게 관리할 수 있다.

트레이드오프:

- 댓글과 마찬가지로 DB foreign key 무결성을 직접 보장하기 어렵다.
- target validation을 애플리케이션 로직에서 책임져야 한다.

추후 개선:

- 동일 사용자 중복 신고 방지 unique 조건 검토
- 신고 사유 enum 또는 코드 테이블 분리
- 처리자 `admin_id` 추가
- 처리 메모, 처리 시간 추가
- 신고 대상 snapshot 저장 검토

### public_data_snapshots

`public_data_snapshots`는 외부 공공데이터 API 응답을 저장한다.

공공데이터는 외부 API 상태, 호출 제한, 응답 변경의 영향을 받는다. 그래서 실시간 호출만 의존하지 않고 지역/데이터 타입별 스냅샷으로 저장할 수 있게 했다.

장점:

- 외부 API 장애나 지연에 덜 의존한다.
- 수집 시점별 데이터를 보관할 수 있다.
- 지역별 공공데이터 대시보드와 동네 비교 기능으로 확장하기 좋다.

트레이드오프:

- `payload_json`을 text로 저장하면 DB 레벨의 세밀한 쿼리는 어렵다.
- 데이터 타입별 정규화가 덜 되어 있어 분석 쿼리에 한계가 있다.

추후 개선:

- PostgreSQL `jsonb` 전환 검토
- 데이터 타입별 전용 테이블 분리
- 수집 실패 로그 저장
- 최신 snapshot 조회용 인덱스 추가
- 수집 주기와 보관 기간 정책 정의

### event_logs

`event_logs`는 서비스 내 주요 이벤트를 저장한다.

게시글 조회, 검색, 필터 사용, 글 작성, 신고, 외부 API 실패 같은 이벤트를 남기면 운영 개선과 장애 분석에 활용할 수 있다.

장점:

- CloudWatch 같은 외부 로그 시스템을 붙이기 전에도 기본 이벤트 추적이 가능하다.
- 기능 사용량과 사용자 행동을 분석할 수 있다.
- API 실패나 이상 행동 추적의 기반이 된다.

트레이드오프:

- DB에 로그를 계속 쌓으면 데이터가 빠르게 증가할 수 있다.
- 실시간 분석용으로는 한계가 있다.

추후 개선:

- CloudWatch 연동
- 이벤트 이름 표준화
- 로그 보관 기간 정책
- 비동기 저장 처리
- 민감 정보 마스킹 정책

## 주요 트레이드오프

### 방 양도글과 커뮤니티 글을 분리한 이유

대안은 모든 게시글을 `posts` 하나로 합치고 `post_type`으로 구분하는 방식이었다.

하지만 방 양도글은 거래성 필드가 많고, 커뮤니티 글은 카테고리와 본문 중심이다. 하나의 테이블에 합치면 nullable 컬럼이 많아지고, 도메인별 검증과 조회 조건이 복잡해질 가능성이 높다.

그래서 초기에는 테이블을 분리하고, 댓글과 신고처럼 공통으로 붙는 기능만 다형 참조로 처리했다.

### 댓글/신고에 다형 참조를 사용한 이유

댓글과 신고를 대상별 테이블로 나누는 방식도 가능했다.

예를 들어 `room_handover_comments`, `community_comments`, `room_handover_reports`, `comment_reports`처럼 나눌 수 있다.

하지만 그러면 API, 관리자 화면, 신고 처리 로직이 중복될 가능성이 높다. MVP에서는 댓글과 신고의 핵심 필드가 대상과 무관하게 거의 같기 때문에 `type + id` 방식으로 공통화했다.

대신 DB foreign key 무결성이 약해지는 단점이 있으므로, 서비스 계층에서 대상 존재 여부 검증을 책임져야 한다.

### 공공데이터를 정규화하지 않고 snapshot으로 둔 이유

초기부터 실거래가, 생활시설, 상권, 치안 등 데이터를 모두 정규화하면 테이블 설계가 커지고 외부 API 변경에 취약해진다.

MVP 단계에서는 먼저 원본 응답을 저장하고, 실제로 자주 쓰는 데이터 타입이 명확해졌을 때 전용 테이블로 분리하는 편이 낫다.

그래서 `source`, `data_type`, `payload_json`, `collected_at` 중심의 snapshot 구조를 선택했다.

### soft delete를 선택한 이유

방 양도글은 `hidden` 컬럼을 통해 삭제된 것처럼 처리한다.

실제 삭제는 구현이 단순하지만, 신고, 운영자 검토, 분쟁 대응에 필요한 이력이 사라진다. 반면 soft delete는 조회 조건 관리가 필요하지만 운영 데이터 보존에 유리하다.

MVP 이후 신고/관리자 기능이 예정되어 있으므로 soft delete가 더 적절하다고 판단했다.

## 인덱스 설계

초기 인덱스는 MVP 조회 패턴을 기준으로 추가했다.

- `idx_oauth_accounts_user_id`: 사용자 기준 OAuth 계정 조회
- `idx_room_handover_posts_region_status`: 지역과 상태 기준 방 양도글 목록 조회
- `idx_room_handover_posts_author_id`: 작성자 기준 방 양도글 조회
- `idx_community_posts_region_category`: 지역과 카테고리 기준 커뮤니티 글 조회
- `idx_comments_post`: 게시글 타입과 게시글 ID 기준 댓글 조회
- `idx_reports_target`: 신고 대상 기준 신고 조회
- `idx_public_data_snapshots_region_type`: 지역과 데이터 타입 기준 공공데이터 조회
- `idx_event_logs_event_created`: 이벤트명과 생성일 기준 로그 조회

추후 실제 쿼리 패턴이 쌓이면 느린 쿼리를 기준으로 인덱스를 조정한다.

## 추후 확장 방향

- 인증 도입 후 작성자 권한 검증 추가
- 게시글 이미지 테이블 추가
- 방 옵션 테이블 분리
- 댓글 soft delete와 대댓글 구조 추가
- 신고 처리자, 처리 메모, 처리 시간 추가
- 공공데이터 `jsonb` 전환 또는 타입별 테이블 분리
- 이벤트 로그 외부 로그 시스템 연동
- 검색 조건이 복잡해지면 QueryDSL 도입
- 관리자 화면을 위한 상태/숨김/audit 컬럼 보강

## 결론

이번 DB 설계는 MVP를 빠르게 구현하면서도 이후 확장 가능성을 열어두는 방향이다.

모든 데이터를 처음부터 완벽하게 정규화하지는 않았다. 특히 댓글/신고의 다형 참조와 공공데이터 snapshot 구조는 빠른 MVP 구현을 위한 선택이다.

대신 서비스의 핵심 축인 사용자, OAuth 계정, 지역, 방 양도글, 커뮤니티 글은 초기에 분리해서 이후 기능 확장 시 큰 구조 변경이 필요하지 않도록 했다.
